## Test Matrix

| Layer       | Owner                                                      | Gate                                                          |
| ----------- | ---------------------------------------------------------- | ------------------------------------------------------------- |
| Format      | Root workspace and all workspaces                          | `bun run format`                                              |
| Lint        | Root config, packages, apps                                | `bun run lint`                                                |
| Typecheck   | All TS packages and apps                                   | `bun run typecheck`                                           |
| Unit        | `packages/domain`, `packages/validation`, pure helpers     | `bun run test` or targeted package tests                      |
| Contract    | `packages/contracts`, `packages/test-utils`, app consumers | `bun run test:contracts`                                      |
| Integration | `apps/product-core` services, APIs, DB, webhooks, cron     | Targeted product-core integration command plus `bun run test` |
| Smoke       | `apps/product-core`, `apps/admin-app`                      | Future `bun run test:smoke` or app-scoped smoke command       |
| E2E         | Cross-app release journeys                                 | Future `bun run e2e`                                          |
| Build       | Buildable apps and packages                                | `bun run build`                                               |

## Phase Regression Policy

Every phase ends with a regression pass using the strongest available command set.

Early phases may lack commands only before the workspace exists. When a command is unavailable, the implementer must state that explicitly, run the closest useful check, and carry the missing command forward. Once Phase 1 creates `package.json`, missing baseline scripts are defects unless the active step documents a temporary no-op.

Regression gates must become stricter as implementation appears. Do not remove, skip, or weaken gates because they are inconvenient.

## Risks

- Phase 1 can accidentally create scripts that pass without checking affected workspaces. The root scripts should route through Turbo and make no-op behavior explicit.
- Contract tests must be added early enough that admin-app cannot drift from product-core API DTOs.
- Billing, auth, and permissions need both domain-level and integration-level coverage; pure UI smoke is not enough.
