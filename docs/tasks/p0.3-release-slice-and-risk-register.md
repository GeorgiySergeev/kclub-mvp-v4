## Context

Step `docs/development/phase-0/step-p0.3-release-slice-and-risk-register.md` requires the MVP release slice and risk register to be defined before scaffold and implementation begin.

P0.1 confirmed the source of truth is the monorepo documentation set. P0.2 confirmed the future Bun/Turbo quality gates and missing-command policy.

## Goal

Keep implementation focused on the production-ready MVP journeys from `docs/SPEC.md` and prevent later phases from drifting into future product scope.

| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | ------------------- | --------------------------------------------- |
| Public visitor discovery           | Visitor can view localized home, directory, business detail, and PII-safe card verification.                                 | Phase 4             | Smoke, E2E, build                             |

## Future Scope Excluded From MVP

Do not implement these in MVP unless Product explicitly changes `docs/SPEC.md`:

- Business profile draft mode.
- Custom invoice UI.
- Public arbitrary card lookup form.
- Guest-only partner cabinet.
- MLM mechanics.
- Payment Links as the billing source of truth.
- Guest partner self-service flows outside member/VIP ownership.

## Risk Register

| Risk                                                                        | Owner phase           | Validation method                                                                                | Mitigation                                                                                                                                           |
| --------------------------------------------------------------------------- | --------------------- | ------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Bun tooling compatibility blocks workspace, Next.js, Turbo, or CI commands. | Phase 1               | `bun install --frozen-lockfile`, root scripts, build/typecheck/lint gates                        | Pin Bun in root `package.json`, keep one `bun.lock`, document any real blocker before switching the monorepo to pnpm.                                |
| Root scripts pass while silently skipping workspaces.                       | Phase 1               | Turbo dry run or equivalent script inspection, package/app smoke once workspaces exist           | Route root scripts through Turbo and make early no-ops explicit.                                                                                     |
| Contract drift between product-core and admin-app.                          | Phases 2, 3, and 6    | `bun run test:contracts`, typecheck both apps on contract/API changes                            | Keep DTOs, errors, permissions, and route contracts in `packages/contracts`; merge product-core API changes with admin-app consumer changes.         |
| Public DTO leaks PII or admin-only fields.                                  | Phases 2, 3, 4, and 7 | Contract tests, DTO mapper tests, public route smoke, security regression                        | Separate public/admin DTOs and add explicit privacy boundary tests for card verification and business detail/list responses.                         |
| Staff auth or TOTP session gating is weak.                                  | Phases 2, 3, 6, and 7 | Domain/permission tests, route guard tests, admin smoke, security regression                     | Enforce staff identity separately from member identity, require TOTP before protected admin routes, and use secure httpOnly sameSite=strict cookies. |
| Stripe webhook idempotency or signature handling fails.                     | Phase 5               | Webhook integration tests, duplicate-event tests, unsigned-request rejection, failure-mode tests | Verify signatures, store processed event IDs, make checkout creation non-authoritative, and publish/activate only from verified webhook events.      |
| Billing state and member capability drift from Stripe.                      | Phase 5               | Subscription policy tests, webhook integration, cron regression, E2E checkout return             | Treat Stripe as source of truth, keep local rows as caches/audit records, and run daily maintenance cleanup.                                         |
| Featured business max-count race condition.                                 | Phases 2, 3, and 6    | Domain policy tests, DB/integration transaction tests, admin smoke                               | Enforce max 3 under transaction/lock and reset featured flags when a business leaves `PUBLISHED`.                                                    |
| Business Introduction rate/cooldown rules are bypassed.                     | Phases 2 and 3        | Domain unit tests, validation tests, API integration tests                                       | Centralize limits in `packages/domain` and enforce server-side before writes.                                                                        |
| Admin-app writes directly to product database.                              | Phases 1, 3, and 6    | Architecture review, import-boundary lint/typecheck, integration smoke                           | Keep admin-app behind product-core admin APIs and proxy/BFF boundary.                                                                                |
| README and docs drift from real scaffold commands.                          | Phase 1               | Manual doc review, markdown link checks, command alignment checks                                | Refresh README after scaffold decisions are materialized and resolve stale `docs/review.md` references from P0.1.                                    |
| Release-critical E2E coverage is deferred too late.                         | Phases 4, 5, 6, and 7 | Smoke and E2E gates per implemented journey                                                      | Add E2E incrementally as journeys become real, then run full critical-path suite before release.                                                     |

## Open Product Defaults

Use these defaults unless Product changes the spec:

- Keep existing `MEM-*` card on MEMBER to VIP upgrade; admin may manually re-issue `VIP-*`.
- Allow only one pending introduction to the same target at a time; apply a 30-day cooldown after terminal state.
- Keep canceled VIP capabilities active until `current_period_end`.
- Do not require business draft mode for MVP launch.
